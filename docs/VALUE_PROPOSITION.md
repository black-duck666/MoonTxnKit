# 项目价值与抽象边界

## 真实问题

许多 MoonBit 工具并不需要完整数据库，却需要同时修改多个内存状态，并保证：

1. 所有前置条件来自同一个一致快照；
2. 一批状态要么全部生效，要么完全不生效；
3. 并发修改不能悄悄覆盖；
4. 提交结果可以记录、校验和恢复；
5. 相同输入在不同编译后端得到相同结果。

如果每个规则引擎、工作流、模拟器和测试替身分别实现这些能力，就会重复出现
版本号、临时副本、补偿代码、锁和日志格式。MoonTxnKit 把这组共性收敛为独立
状态事务内核。

## 合理抽象

`AtomicPlan` 是面向上层的最小抽象：

- condition 描述提交前必须成立的业务事实；
- write 描述必须原子生效的状态集合；
- isolation 决定是否在提交时重新验证读集；
- `PlanResult` 明确区分业务条件失败与并发提交冲突。

它不要求上层理解版本链或 WAL，也不把业务对象强制建模成数据库表。键和值
目前采用 String，允许调用方自行选择 JSON、领域编码或标识符约定。

## 复用对象

### 工作流与任务调度

只有 `state=queued` 且 owner 不存在时才能领取任务。多个 worker 同时竞争时，
最多一个计划提交成功，失败者获得可解释结果。

### 库存与资源预占

库存版本、订单状态和幂等标记可以在同一快照检查，库存扣减、订单迁移和预占
记录使用同一个提交版本落地，不会产生半完成状态。

### 规则引擎

规则计算读取一组事实，Serializable 读集校验保证规则结论提交前，这些事实
没有被其他事务改变。

### 模拟器和测试替身

逻辑版本、结构化冲突和纯内存 WAL 不依赖线程调度或系统时钟，适合重放测试、
差异测试和教学可视化。

### 持久化适配器

核心生成可校验的逻辑 WAL。上层可以把记录写入文件、浏览器存储、对象存储或
数据库，并在启动时恢复，而无需改变事务语义。

## 非目标

MoonTxnKit 不实现 SQL、索引、查询优化、文件刷盘、网络复制和分布式共识。
它也不宣称替代成熟数据库。项目价值在于为“不值得引入完整数据库、但不能
接受部分状态和不可恢复更新”的 MoonBit 组件提供中间层。
